# 일반 RAG vs Graph RAG: 핵심 차이점 비교

## 1. 근본적인 차이

### 1.1 한 문장 요약

| 구분 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **핵심 개념** | "비슷한 텍스트 찾기" | "연결된 지식 탐색하기" |
| **비유** | 키워드로 책 찾는 검색 엔진 | 개념 간 관계를 보여주는 마인드맵 |

### 1.2 아키텍처 비교

```
[일반 RAG]
문서 → 청크 → 임베딩 → 벡터 DB
                         ↓
질문 → 임베딩 → 유사도 검색 → Top-K 청크 → LLM → 답변

[Graph RAG]
문서 → 엔티티/관계 추출 → 지식 그래프 → 커뮤니티 탐지 → 요약 생성
                                              ↓
질문 → 그래프 쿼리 → 관련 노드/경로 탐색 → 구조화된 컨텍스트 → LLM → 답변
```

## 2. 핵심 구성요소 비교

### 2.1 지식 표현 방식

| 측면 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **데이터 구조** | 평면적 문서 구조 (Flat) | 그래프 구조 (노드 + 엣지) |
| **저장 방식** | 텍스트 청크의 벡터 임베딩 | 엔티티(노드)와 관계(엣지) |
| **맥락 표현** | 청크 내 지역적 맥락만 보존 | 전역적 관계 맥락 보존 |

**시각적 예시:**

```
[일반 RAG의 데이터]
┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐
│ Chunk 1         │  │ Chunk 2         │  │ Chunk 3         │
│ "삼성전자는..."  │  │ "이재용 회장은.."│  │ "반도체 사업..."│
│ Vector: [0.2..] │  │ Vector: [0.3..] │  │ Vector: [0.1..] │
└─────────────────┘  └─────────────────┘  └─────────────────┘
     (각각 독립적으로 존재)

[Graph RAG의 데이터]
    ┌──────────┐     CEO of      ┌──────────┐
    │ 삼성전자  │◄───────────────│ 이재용    │
    └────┬─────┘                 └──────────┘
         │ operates
         ▼
    ┌──────────┐
    │ 반도체    │
    │ 사업부    │
    └──────────┘
     (관계로 연결됨)
```

### 2.2 검색 메커니즘

| 측면 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **검색 방식** | 벡터 유사도 검색 | 그래프 순회 알고리즘 |
| **쿼리 처리** | 단일 벡터 인덱스 쿼리 | 다단계: 쿼리 해석 → 서브그래프 선택 → 그래프 쿼리(Cypher 등) |
| **검색 깊이** | Single-hop (직접 유사도) | Multi-hop (관계 체인 따라 탐색) |

## 3. 시나리오별 성능 비교

### 3.1 Multi-hop 추론이 필요한 경우

**시나리오:** "A 회사의 CEO가 투자한 스타트업 중 AI 기술을 보유한 곳은?"

```
[일반 RAG]
- 검색 결과:
  - Chunk A: "A 회사 CEO는 김철수"
  - Chunk B: "김철수는 B, C, D 스타트업에 투자"
  - Chunk C: "B 스타트업은 핀테크 기업"
  - Chunk D: "C 스타트업은 AI 기술 보유"
- 문제: 청크들이 개별적으로 검색됨. LLM이 관계를 추론해야 함
- 결과: 불완전하거나 부정확한 답변 가능성 높음

[Graph RAG]
- 그래프 탐색:
  A회사 --[CEO]--> 김철수 --[투자]--> [B, C, D] --[기술]--> AI
- 경로 추적으로 C 스타트업 직접 도출
- 결과: 정확하고 설명 가능한 답변
```

### 3.2 전역적 요약이 필요한 경우

**시나리오:** "이 문서 컬렉션의 주요 테마와 인사이트는?"

```
[일반 RAG]
- 개별 청크에서 유사한 내용 검색
- 전체적인 패턴이나 테마 파악 어려움
- 일부 관련 청크만 반환

[Graph RAG]
- 커뮤니티 구조 분석으로 주요 테마 그룹 식별
- 계층적 요약(Level 0: 최상위 테마, Level 1: 세부 주제)
- 전역적 관점의 인사이트 제공
```

### 3.3 정보 흩어짐 문제

**시나리오:** "CEO가 Q4와 Q1 보고서의 인사이트를 바탕으로 도입한 정책은?"

| 일반 RAG 결과 | Graph RAG 결과 |
|--------------|----------------|
| Q4 보고서 청크와 Q1 보고서 청크가 분리 검색 | Q4 인사이트 → CEO 결정 → 새 정책 경로 추적 |
| 관계 추론은 LLM에게 위임 | 명시적 관계 기반 컨텍스트 제공 |
| 정보 연결 실패 가능성 높음 | 연결된 정보를 구조화하여 제공 |

## 4. 벤치마크 성능 (2025)

### 4.1 정량적 비교

| 메트릭 | 일반 RAG | Graph RAG | 개선율 |
|--------|----------|-----------|--------|
| **RobustQA 평균 점수** | 59-72% | 86.31% | ~20-45% 향상 |
| **비즈니스 질문 정확도** (Data.world) | 기준선 | 3배 향상 | 200% 향상 |

> "Graph RAG는 RobustQA 평균 점수 86.31%를 달성하여 다른 접근법을 상당한 차이로 앞섰습니다. 비교적으로 Azure Cognitive Search + GPT-4는 72.36%, 다른 벡터 기반 방법은 59.61%~69.02%를 기록했습니다."
> — [arXiv 2502.11371](https://arxiv.org/abs/2502.11371)

### 4.2 작업 유형별 강점

| 작업 유형 | 일반 RAG | Graph RAG |
|----------|----------|-----------|
| **단순 사실 질문** | 우수 | 우수 |
| **복잡한 관계 질문** | 부족 | 우수 |
| **다중 문서 종합** | 부족 | 우수 |
| **전역 요약** | 부족 | 우수 |
| **실시간 업데이트** | 우수 | 부족 |

## 5. 설명 가능성 (Explainability)

### 5.1 일반 RAG의 한계

```
질문: "왜 이 답변이 나왔나요?"
일반 RAG: "유사도가 높은 청크 3개를 검색했습니다"
         → 왜 그 청크가 검색되었는지 추적 어려움
```

### 5.2 Graph RAG의 강점

```
질문: "왜 이 답변이 나왔나요?"
Graph RAG:
  "다음 경로를 통해 답변을 도출했습니다:
   [회사A] --투자--> [스타트업B] --개발--> [AI기술]
   신뢰도: 0.95"
         → 추론 과정을 명확히 추적 가능
```

| 측면 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **답변 근거** | 유사 청크 목록 | 추적 가능한 노드/경로 |
| **디버깅** | 어려움 | 용이함 |
| **규제 준수** | 제한적 | 감사 추적 가능 |

## 6. 비용과 복잡성 트레이드오프

### 6.1 인덱싱 비용

| 항목 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **초기 인덱싱** | 임베딩 생성 (상대적 저렴) | 엔티티 추출 + 관계 매핑 + 커뮤니티 탐지 + 요약 생성 (고비용) |
| **업데이트** | 새 문서만 임베딩 추가 | 전체 리인덱싱 필요할 수 있음 |
| **토큰 비용** | 낮음 | 상당히 높음 |

### 6.2 LazyGraphRAG (Microsoft의 해결책)

- **문제**: 기존 GraphRAG의 높은 초기 인덱싱 비용
- **해결**: 사전 요약 없이 지연(lazy) 처리
- **비용**: 벡터 RAG와 동일한 인덱싱 비용, 기존 GraphRAG의 0.1%

### 6.3 구현 복잡성

| 측면 | 일반 RAG | Graph RAG |
|------|----------|-----------|
| **기술 스택** | 벡터 DB + LLM | 그래프 DB + 벡터 DB + LLM + NLP |
| **전문성 요구** | 상대적 낮음 | 그래프 이론, 온톨로지 설계 필요 |
| **유지보수** | 단순 | 스키마 진화, 엔티티 해소 관리 |

## 7. 언제 무엇을 선택할까?

### 7.1 일반 RAG 선택

- 단순한 Q&A 시스템
- 비구조화된 텍스트 중심 데이터
- 빠른 프로토타이핑 필요
- 실시간 업데이트가 빈번
- 비용 제약이 있는 경우

### 7.2 Graph RAG 선택

- 복잡한 관계 추론 필요
- 다중 홉(Multi-hop) 질문 지원
- 설명 가능성이 중요 (금융, 의료, 법률)
- 대규모 문서 컬렉션의 전역적 이해
- 엔터프라이즈 지식 관리

### 7.3 하이브리드 접근 (권장)

```
[권장 전략]
1. 벡터 검색으로 시작 (넓은 시맨틱 검색)
2. 그래프 탐색으로 정제 (관계 기반 필터링)
3. 결합된 컨텍스트로 LLM 생성
```

## 8. 알려진 한계점

### 8.1 Graph RAG의 현재 한계

| 한계 | 설명 |
|------|------|
| **엔티티 해소** | 같은 이름의 다른 엔티티 구분 어려움 (동명이인 문제) |
| **정적 요약** | 새 데이터 추가 시 전체 리인덱싱 필요할 수 있음 |
| **확장성** | 대규모 데이터셋에서 성능 이슈 가능 |
| **데이터 프라이버시** | 그래프 구조가 민감 정보 관계 노출 가능 |

### 8.2 2025년 연구 동향

2025년 2월 arXiv 논문에서 RAG와 GraphRAG를 Q&A 및 쿼리 기반 요약 등 벤치마크 작업에서 체계적으로 평가했습니다. 결과는 다양한 작업과 평가 관점에서 RAG와 GraphRAG의 뚜렷한 강점을 강조합니다.
